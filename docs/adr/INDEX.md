# INDEX — ADRs del ERP SaaS Multi-Tenant

> **Criterio rector del producto:** [`docs/FUNDAMENTOS-Y-VISION.md`](../FUNDAMENTOS-Y-VISION.md) — visión (multi-tenant estilo SAP Public Cloud, un Core, cada cliente un tenant), modelo de tenants, plugins y restricciones de plataforma. Marco dentro del cual estos ADR son válidos; leer **antes** de decidir.

Punto de entrada para cualquier sesión nueva con Claude. Pegá este índice primero; el detalle completo de cada ADR se carga solo si hace falta.

| ADR | Decisión | Resumen en una línea |
|---|---|---|
| 001 | Multi-tenant | Shared schema + `tenant_id` + Row-Level Security de Postgres. Camino de escape a schema/DB dedicado solo para enterprise puntuales. |
| 002 | Core / Blueprint / Plugin | Blueprints = configuración pura, cero schema propio. Plugins se comunican por eventos asíncronos con outbox pattern, nunca acceso directo a datos del Core. |
| 003 | Business Capabilities del piloto "Servicios" | Solo Scheduling es capability nueva; todo el resto (Party, Producto, Orden, Pago, Factura) es Core reutilizado. Profesional ≠ Usuario. |
| 004 | Modelo de datos de Scheduling | Overbooking se previene con `EXCLUDE USING GIST` de Postgres, no con lógica de aplicación. |
| 005 | Stack técnico | Postgres (Neon/Supabase → RDS) + NestJS/TypeScript + Next.js + pg-boss + Redis (Upstash) + R2 + Railway/Render → AWS Fargate a escala. Auth propio, no vendor. |
| 006 | Motores de plataforma | De 8 motores propuestos, 4 se construyen para el MVP (Metadata, Workflow básico, Feature Flags, Integration/Plugin) y 4 se difieren o simplifican (Rules como config plana, Tax sobre Rules, AI como capa delgada, Marketplace afuera). |
| 007 | Análisis financiero | ~$5/mes a 1 cliente, ~$300/mes a 100, ~$1.500-2.700/mes a 1.000. Punto de decisión de réplicas/particionado entre 500-2.000 tenants activos. |
| 008 | Costo de tokens de Claude | Un thread por tema, decisiones persistidas como ADR, índice liviano como entrada, modelo barato para tareas mecánicas, Claude Code para trabajar sobre el repo real. |
| 009 | UX, UI metadata-driven, RBAC, onboarding | Diseño para la recepcionista, no para el ERP: agenda como home, turno en <30 seg, mobile-first operativo. DynamicForm renderiza campos de extensión. RBAC de 3 roles por Blueprint. Audit trail desde Fase 1. Onboarding con wizard + importador CSV. |
| AMD | Enmiendas a 001-008 | Restore por tenant probado, soft-delete, versionado+idempotencia de eventos, notas y precio congelado en Turno, horarios/ausencias de profesional, MFA+rate limit en login, costo de email transaccional. |

Email real (Resend); WhatsApp simulado hasta conectar proveedor. **Documentado retroactivamente 2026-07-03** (el código lo citaba sin ADR escrito). Implementado. |

| 018 | Activación de RLS de Postgres | Fija el mecanismo y el momento del backstop que ADR-001 dejó en abstracto: RLS por `tenant_id` con `SET LOCAL app.current_tenant_id` por transacción (pooling-safe, `tenantId` es texto/cuid) vía extensión de Prisma + rol de app sin `BYPASSRLS`; activación como **gate duro previo al 2º tenant**, ensayada en branch de Neon. No adelanta RLS (el diferimiento sigue); le da el "qué sigue" al throw de ADR-015. Pendiente de implementación (disparo: alta del tenant #2). |
| 019 | Onboarding / alta de tenant (provisioning) | Baja la visión self-service de ADR-009 §5 a la Fase 1 real: el alta se hace con un **script operado, idempotente por `slug`, transaccional** (`scripts/provision-tenant.ts`) que siembra tenant + OWNER (patrón ADR-017) + catálogo blueprint mínimo editable (nunca vacío) — **no** portal público ni panel super-admin ni importador CSV (los tres diferidos hasta que las altas frecuentes los justifiquen). El alta del 2º tenant es un **gate compuesto con ADR-018**: son el mismo trabajo (crear la 2ª fila en `Tenant` rompe `getCurrentTenantId()` sin RLS). No implementa código. Script **escrito en la rama `feature/performance`** (pendiente de verificación + merge; disparo real: 2º cliente). |
| 020 | Contrato de API pública del Core | Nombra la "API pública" que ADR-002/006 dejaron abstracta: las ~90 Server Actions por dominio **son** el contrato. Fija convención comando/consulta, propiedad de tablas por capability y la regla de cruce de límites (el cruce Scheduling→`Product.stock` de `completeAppointment` queda nombrado y encauzado). No agrega capa/CQRS/bus: describe lo que ya existe y reserva lo event-driven (outbox) para el 1er Plugin (ARCA). No implementa código. |
| 021 | Consola de operación / super-admin | Scopea el panel super-admin que ADR-019 difirió: es un **plano de plataforma separado** (cross-tenant, otra audiencia y otro plano de autorización que `/admin`), que por seguridad **no** debe pegarse a la app del tenant ni reusar su sesión/rol de DB. Choca de frente con RLS (ADR-018): su implementación es una `/sesion-feature` posterior, atada al gate del 2º tenant. No implementa código. Pendiente de implementación. |
| 023 | Check de performance multi-tenant + restricciones free plan | Los cimientos de datos están bien (índices que lideran con `tenantId`), pero las queries de agenda/dashboard/reportes **no filtran por `tenantId`** → los índices compuestos no rinden a escala (F1): se resuelve activando RLS (ADR-018), que además de aislar **inyecta el predicado** y enciende esos índices — RLS pasa a ser también palanca de performance. Overbooking (F2): ADR-004 promete `EXCLUDE USING GIST` pero el código usa check-then-insert en ReadCommitted → carrera TOCTOU de doble reserva **hoy**; fix Serializable ya, GIST a plan pago, y se **enmienda ADR-004**. Reportes agrega todo el histórico en JS (F3→`groupBy`), N+1/over-fetch menores (F4/F5). Free plan de Neon como techo de piloto: `connection_limit` bajo + validar RLS sobre pooler (F6), `getCurrentTenantId` sin cache se deja (F7, se autoliquida con RLS), y **el storage 0.5 GB es el primer techo** — `AuditLog` append-only necesita retención (F8, enmienda ADR-009/ADR-007). Check 100% estático (cero queries a prod). **No implementa código** — implementación por `/sesion-feature` según §4. |
| 022 | Plugin ARCA (facturación electrónica) | `arca` **no** es producto aparte: es el **primer Plugin** del Core. Vive en `src/plugins/arca/`, hace **solo autorización fiscal** (CAE vía WSAA+WSFEv1), habla con el Core únicamente por el evento `InvoiceCreated` (entra) y el comando `RegisterFiscalDocument` (sale) —ambos con `tenantId`, declarados en su manifiesto (ADR-006)—. El cálculo de IVA queda en el Core (ADR-006). Instancia las superficies II/III que ADR-020 reservó. **Scaffold del plugin entregado contra un stub (cero DB)**; el lado Core (modelo `Invoice` + outbox + worker + comando + credenciales por tenant) se difiere a una `/sesion-feature` con migración y gate. |

## Estado del proyecto (actualizado 2026-07-04)
- **Piloto vivo:** `estetica-erp` (este repo) — Beauty & Spa "CH Estética" de Carolina Haponiuk, desplegado en Netlify + Postgres (Neon). El auto-publish de Netlify está **apagado** (`stop_builds`): el push a `main` va a GitHub sin publicar; el deploy a producción es un gate manual (ver `docs/METODO-ROLES.md` §4).
- **Decisión de convergencia:** ADR-010, **Camino A confirmado** (evolucionar el piloto Next.js, no reescribir) — ya no está bloqueada, esto quedó viejo en una versión anterior de este índice.
- **G1 (tenant_id + RLS), estado real verificado en `prisma/schema.prisma`:** la columna `tenantId` ya está en toda tabla de negocio y toda query pasa por `getCurrentTenantId()` (aislamiento a nivel de aplicación, funcionando). La mitad que falta es **RLS de Postgres** — diferida a propósito (ver comentario en el modelo `Tenant` del schema) hasta que exista un segundo tenant real, porque con uno solo la fuga no es posible **siempre que la resolución de tenant sea fail-closed** (ADR-015: `getCurrentTenantId()` lanza error si hubiera ≠1 tenant en vez de agarrar "el más viejo" en silencio — implementado, sin cache, en `src/lib/tenant.ts`). Activar RLS+pooling antes es el tramo más riesgoso de tocar sin necesidad. **Novedad (ADR-018):** ya no está "sin plan" — el *mecanismo* (RLS + `SET LOCAL` por transacción vía extensión de Prisma + rol sin `BYPASSRLS`) y el *momento* (gate duro previo al 2º tenant, ensayado en branch de Neon) quedaron decididos; falta solo implementarlo el día del gate. **No asumir que esto ya está cerrado** — la implementación sigue pendiente; es el primer punto a verificar en cualquier sesión nueva.
- **Ola 2 (ADR-011):** completa — G16/G9/G17/G18 implementados y deployados.
- **Brechas nuevas del relevamiento contra la competencia (2026-07-03):** ADR-013 (G19) y ADR-014 (G20 parcial, G21 completo) — ver arriba.
- **Backlog operativo vivo (features de negocio, no arquitectura):** `BACKLOG.md` en la raíz del repo — auditado contra el código, no contra supuestos. Roles de usuario y reprogramar turno del cliente son los ítems críticos abiertos ahí.

# INDEX — ADRs del ERP SaaS Multi-Tenant

> **Criterio rector del producto:** [`docs/FUNDAMENTOS-Y-VISION.md`](../FUNDAMENTOS-Y-VISION.md) — visión (multi-tenant estilo SAP Public Cloud, un Core, cada cliente un tenant), modelo de tenants, plugins y restricciones de plataforma. Marco dentro del cual estos ADR son válidos; leer **antes** de decidir.

Punto de entrada para cualquier sesión nueva con Claude. Pegá este índice primero; el detalle completo de cada ADR se carga solo si hace falta.

| 020 | Contrato de API pública del Core | Nombra la "API pública" que ADR-002/006 dejaron abstracta: las ~90 Server Actions por dominio **son** el contrato. | 023 | Check de performance multi-tenant + restricciones free plan | Los cimientos de datos están bien (índices que lideran con `tenantId`), pero las queries de agenda/dashboard/reportes **no filtran por `tenantId`** → los índices compuestos no rinden a escala (F1): se resuelve activando RLS (ADR-018), que además de aislar **inyecta el predicado** y enciende esos índices — RLS pasa a ser también palanca de performance. Overbooking (F2): ADR-004 promete `EXCLUDE USING GIST` pero el código usa check-then-insert en ReadCommitted → carrera TOCTOU de doble reserva **hoy**; fix Serializable ya, GIST a plan pago, y se **enmienda ADR-004**. Reportes agrega todo el histórico en JS (F3→`groupBy`), N+1/over-fetch menores (F4/F5). Free plan de Neon como techo de piloto: `connection_limit` bajo + validar RLS sobre pooler (F6), `getCurrentTenantId` sin cache se deja (F7, se autoliquida con RLS), y **el storage 0.5 GB es el primer techo** — `AuditLog` append-only necesita retención (F8, enmienda ADR-009/ADR-007). Check 100% estático (cero queries a prod). **No implementa código** — implementación por `/sesion-feature` según §4. |
| 022 | Plugin ARCA (facturación electrónica) | `arca` **no** es producto aparte: es el **primer Plugin** del Core. Vive en `src/plugins/arca/`, hace **solo autorización fiscal** (CAE vía WSAA+WSFEv1), habla con el Core únicamente por el evento `InvoiceCreated` (entra) y el comando `RegisterFiscalDocument` (sale) —ambos con `tenantId`, declarados en su manifiesto (ADR-006)—. El cálculo de IVA queda en el Core (ADR-006). Instancia las superficies II/III que ADR-020 reservó. **Scaffold del plugin entregado contra un stub (cero DB)**; el lado Core (modelo `Invoice` + outbox + worker + comando + credenciales por tenant) se difiere a una `/sesion-feature` con migración y gate. |
| 024 | Disparadores de facturación + toggle + Plugin Mercado Pago | Completar un servicio (`completeAppointment`) **dispara la facturación** ARCA (ADR-022), con un **toggle *facturar sí/no*** en el cierre (default sí; MP siempre auto-factura). Todo detrás de un **feature flag OFF por default** (`ARCA_INVOICING_ENABLED`) porque la migración de `Invoice` no está aplicada: en prod hoy `completeAppointment` no cambia. **Mercado Pago** entra como 2º plugin hexagonal (`src/plugins/mercadopago/`, port + stub + webhook handler): pago acreditado → auto-factura. El **Core es dueño del impuesto** (`src/lib/fiscal.ts`, semilla Tax Engine); perfil fiscal provisional (ch estética = Monotributo→Factura C). Implementado con stubs, cero DB/red; adapters reales + credenciales por tenant, diferidos. |
| 025 | Ingesta de Mercado Pago + facturación masiva (producto monotributista) | Amplía ADR-024: MP no es "facturar un pago suelto" sino **detectar TODO lo que le entra a un comerciante por MP y facturárselo solo** — **producto** para monotributistas/comercios chicos con muchas operaciones (arca standalone, además de plugin). Diseño: **dos fuentes convergentes** (backfill histórico paginado + webhook), **idempotencia por `payment_id`** vía registro de conciliación (que es también el estado del producto), **outbox+worker con rate-limit** hacia ARCA para volumen, **una Factura C por operación** por default (agrupado opcional futuro). **Acceso por OAuth de MP** (authorization code; NUNCA scraping — frágil/contra ToS/no escala, §9). **Modelo "contador socio"**: un operador administra por OAuth la **cartera** de monotributistas, cada cliente como **tenant aislado** + operador cross-tenant (patrón ADR-021/019, §10). **Diferencial** vs Facturitas (manual)/Facturante (solo posnet)/TusFacturasApp·iFactura (cobro de facturas): **ingesta automática de TODO el feed MP + multi-cliente por el contador** (§11). **Solución COMPLETA end-to-end** (§12): OAuth → ingesta → **clasificación de ingresos** (facturable/no/revisar — MVP, prevención fiscal: NO facturar transferencias/devoluciones/préstamos) → facturación ARCA → conciliación → **panel del contador** (v1+, canal de distribución) → **alerta de recategorización de monotributo** (v1+) → **confirmación por WhatsApp** (v1+, port de notif). Núcleo con stubs: `sincronizarPagos` (ingesta idempotente + clasificador cableado), `ClasificadorPorReglas`, `NotificacionWhatsAppStub`, `facturarPagoMP`, feed simulado. Smoke: 7 ops mixtas clasificadas OK + 120 ops sin duplicar. OAuth real + panel + alertas + WhatsApp real + tabla de conciliación + worker, diferidos por roadmap (MVP/v1+/visión). |

| 026 | Harness de tests — `node:test` + `tsx` | El repo no tenía **ningún test automatizado**; la verificación era solo `tsc`+build+preview, que no protege la **lógica de dominio** de regresiones. Se adopta el runner **built-in `node:test`** corrido con **`tsx`** (`npm test` → `node --import tsx --test "src/**/*.test.ts"`, glob acotado a `src/` para no correr los smoke scripts de `scripts/` que tocan la DB): **cero dependencias nuevas** (alineado con la economía de deps de ADR-008), TS sin build, tests **colocados al lado del código** que prueban **lógica pura o con deps inyectadas/mockeadas** (mismo patrón de `provisionTenant`/`purgeAuditLogs`), **sin tocar la DB**. Primeras pruebas: `audit-retention` (ventana + purga con mock) y `report-config` (invariantes). No es gate de deploy. Prioridad: cubrir la lógica de mayor riesgo (reserva/fiscal/retención), no perseguir % de cobertura. |
| 027 | Analytics cross-tenant — benchmarking anónimo por rubro | Habilita la **palanca #1 del sector Agencia** (analytics-producto) **sin romper el aislamiento**: el plano del tenant (app + RLS, ADR-018) **nunca** lee otro tenant; el valor cross-tenant lo produce un **pipeline de agregación de plataforma** (plano ADR-021, rol de DB propio, sobre branch/réplica, no prod) que emite **solo agregados anónimos por cohorte** (rubro×zona×período) a una tabla de benchmarks sin `tenantId` de terceros. Privacidad: **k-anonymity k≥5** + supresión si `n<k` + anti-dominancia (l-diversity/t-closeness) + opt-in; el tenant ve *"vos vs. la cohorte"*, nunca *"vs. negocio X"*. **Gate de masa:** sin sentido hasta ≥5 tenants/cohorte (hoy 2 → se diseña, se activa post-escala). El analytics **single-tenant** ("Panel del Dueño") **no** depende de este ADR y va ya. No implementa código; tabla nueva ⇒ migración = Gate 2. |
